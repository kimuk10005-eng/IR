3차 개선본 피드백 / 결론 요약

1. 이번 구조의 핵심 방향
- 목적은 다중 액체 분류가 아니라 액체 / 바닥 2진 감지로 단순화하는 것이다.
- 학습 데이터와 예측 데이터 모두 같은 root 경로 기준으로 연결되도록 정리하였다.
- VS Code에서 따로 실행할 수 있도록 train 코드와 predict 코드를 분리하였다.

2. 기존 구조에서 가장 큰 문제
- 기존 방식은 global baseline 기준 변화량에 많이 의존했다.
- 그런데 실제 데이터에서는 바닥 파일이어도 전체 신호가 위나 아래로 떠 있는 경우가 존재한다.
- 그래서 바닥인데도 액체처럼 보이는 window가 생기고, 이게 학습되면 과탐지가 증가한다.

3. 이번 개선 방향
- 전역 baseline보다 window 직전의 최근 안정구간을 local baseline으로 사용했다.
- 즉 액체 여부를 "처음 baseline에서 얼마나 멀어졌는가"보다 "직전 안정구간에서 얼마나 달라졌는가"로 보도록 바꾸었다.
- 바닥 데이터 중 local baseline 기준 변화가 너무 큰 구간은 오염 구간으로 보고 일부 제외하였다.
- 액체 데이터도 변화가 매우 약한 구간은 과도하게 반복 학습되지 않도록 증강량을 줄였다.

4. 학습 코드와 예측 코드 구성
- liquid_floor_train_v4.py
  : floor / liquid 데이터를 읽어서 바닥별 모델을 학습하고 저장한다.
- liquid_floor_predict_v4.py
  : 저장된 모델을 그대로 불러와 scenario_predict 폴더의 csv를 예측한다.

5. 경로 구조
- root\trimmed_manual_split\floor
- root\trimmed_manual_split\liquid
- root\kumoh_binary_model_save
- root\scenario_predict
- root\predict_results_binary

6. 기대 효과
- 바닥을 액체로 잘못 감지하는 과탐지는 기존보다 줄어들 가능성이 높다.
- 특히 baseline 전체가 흔들린 바닥 데이터에 대해 더 안정적일 가능성이 있다.
- 학습과 예측이 같은 root 경로를 쓰므로 경로 오류가 줄어든다.

7. 아직 남는 한계
- local baseline으로 바꿔도 액체 변화가 매우 약하면 미탐 가능성은 남는다.
- 데이터 자체가 서로 너무 비슷하면 모델 구조를 바꿔도 구분 한계가 있다.
- 따라서 성능이 부족할 경우 가장 먼저 봐야 할 것은 모델보다 학습 데이터 품질과 라벨 구간이다.

8. 최종 결론
- 지금 단계에서는 구조를 더 복잡하게 추가하는 것보다
  global baseline 의존을 줄이고 local baseline 중심으로 단순화하는 것이 더 타당하다.
- 연구 결론도 "복잡한 후처리 추가"보다 "학습 데이터 구간 선정과 기준 baseline 정의가 성능에 더 큰 영향"이라고 정리하는 것이 맞다.
